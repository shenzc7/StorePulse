1. Introduction

1.1 Scenario and Operational Use Cases

The central environmental scenario defining the StorePulse ecosystem concerns modern retail management (encompassing physical storefronts, e-commerce distribution centers, and mid-tier supply chain integrators). The primary systemic expected use cases are modeled around user entities actively engaging localized data ingestion endpoints via the React frontend to derive predictive logistics:

  • Automated Predictive Demand Forecasting: Producing rigorous calculations surrounding future inventory demands by algorithmically parsing deep historical sales datasets without mandating localized Data Science expertise from the end user.
  • Bifurcated Functional Input Tiers (Lite vs. Pro Modes): The system deliberately segregates data complexities. "Lite Mode" accommodates low-density CSV or manual array inputs containing isolated variable metrics (e.g., purely Date and Total Sales). Conversely, "Pro Mode" facilitates robust, multi-variate multi-dimensional data uploads encompassing extraneous variables like marketing spend, category divisions, and localized pricing elasticity indices.
  • Algorithmic UI Summarization Layer: Bridging the calculation layer directly into instantaneous, human-readable semantic business insights via the `IntelligentInsights.tsx` component hierarchy, translating raw percentage variations into direct managerial instructions.

1.2 Boundary Contexts and System Limitations

While highly optimized, StorePulse operates under explicitly defined mathematical and environmental boundary limitations that must be acknowledged to define output confidence accurately:

  1. Exogenous Event Opacity (Black Swans): Internal autoregressive models depend fundamentally on historical cyclicality. Without integrated real-time external APIs feeding localized topological conditions (e.g., severe localized weather events, hyper-regional cultural events), the model may falsely predict linear continuations inside non-linear event horizons.
  2. Input Matrix Sparsity: Early deployment instances utilizing the "Lite Mode" suffer algorithmic cold-start limitations. Inadequate historical temporal depth mathematically degrades the error confidence ceilings generated by back-propagating solvers.
  3. State Management Overheads: Overly vast continuous array inputs passed from the Python server logic occasionally cause Document Object Model (DOM) repainting stutters within React if not rigidly memoized via the useMemo hook configurations inside `reportsPage.tsx`.

1.3 Problem Statement Formulation

The current commercial software paradigm presents a severe disconnect for mid-market business operators. Managers are relegated to utilizing either computationally rigid, offline spreadsheet macros devoid of machine-learning forecasting capabilities, or alternatively, utilizing immensely overly-complex ERP applications. These massive systems function as opaque black boxes mapping generic algorithms without offering the manager specific semantic reasoning for the generated forecast arrays.

The software must successfully automate asynchronous data array ingestion, apply mathematically rigid normalization parameters, generate explicit NB-INGARCH time series predictions, and seamlessly translate the finalized analytical schema back to localized responsive UI dashboards.

3.3 Backend API Router & Advanced Forecast Logic (`api/routes/forecast.py`)

The backend ecosystem is driven completely by route structures instantiating temporary DataFrames. The pipeline enforces rigorous schema typing utilizing Pydantic models before processing. `api/routes/data.py` primarily sanitizes categorical inputs from the `Pro Mode` payloads executing one-hot mapping structures where necessary.

At the heart of `api/routes/forecast.py` exists the Negative Binomial Integer-valued GARCH (NB-INGARCH) equation, tasked with generating the sequential forward steps mathematically mapping $Y_t$. Time series relies on regressing variables against their own previous time-lagged values, fundamentally mapped for integer counts.

$$ P(Y_t = y | \mathcal{F}_{t-1}) = \frac{\Gamma(y + \nu)}{y! \Gamma(\nu)} p_t^\nu (1-p_t)^y $$ $$ \lambda_t = \beta_0 + \sum_{i=1}^{p} \alpha_i Y_{t-i} + \sum_{j=1}^{q} \beta_j \lambda_{t-j} $$
Equation 3.1: Complete structural representation of the NB-INGARCH regression pipeline executed localized inside Python core.

Here, the conditional mean predicting future events is $\lambda_t$, mathematically influenced by the constant base scalar $\beta_0$. The explicitly mapped historical observations (defined up to order $p$) use trained coefficient weights denoted by $\alpha_i$ to explicitly model autoregressive dynamics, capturing count dependencies. The error distribution is captured by the structural dispersion parameter $\nu$ mapping overdispersion relative to the conditional variance ($\lambda_t + \lambda_t^2/\nu$) smoothed by historical means $\beta_j$. Finally, the integer constraint binds probability $P$.

4.3 Data Imputation Bridging Strategies

When missing weekly timeline objects inevitably occur ($NaN$ arrays), Python explicitly executes localized interpolations using the Simple Moving Average (SMA) sliding window technique bounded by $k$ structural lags:

$$ Y_{t}^{(imputed)} = \frac{1}{k} \sum_{j=1}^{k} Y_{t-j} \quad \Big| \quad Y_t = \text{NaN} $$

4.4 Enforcing Stationarity: The Augmented Dickey-Fuller (ADF) Test

Time-series modeling assumes metrics do not inherently drift eternally via vast historical trends without structural boundaries. If $Y_t$ lacks stationarity (exhibiting constant upward mean drift), predictions will decay completely. The backend evaluates stationarity explicitly via ADF testing algorithms. The hypothesis asserts the presence of a unit root natively inside the mathematical representation:

$$ \Delta Y_t = \alpha + \beta t + \gamma Y_{t-1} + \sum_{i=1}^{p} \delta_i \Delta Y_{t-i} + \varepsilon_t $$
Equation 4.4: Augmented Dickey-Fuller equation checking for unit root ($\gamma = 0$). If tests fail, differing matrix operators apply.

5.2 Logic Validation and Accuracy Metrics (MAPE & RMSE)

Rigorous algorithmic assertion testing actively partitions raw payload arrays dynamically creating "Training vs Validation" splits without manual intervention. To intelligently inform `SmartSummary.tsx` on exactly which textual insight output array to fetch, explicit absolute loss variables must be structurally generated natively within Python execution bounds.

The project universally implemented Mean Absolute Percentage Error (MAPE) to generate an easily digestible percentage boundary passed to the React UI component.

$$ \text{MAPE} = \left( \frac{100\%}{n} \right) \sum_{t=1}^{n} \left| \frac{Y_{\text{actual}, t} - Y_{\text{forecast}, t}}{Y_{\text{actual}, t}} \right| $$
Equation 5.1: Scale-independent percentage scoring driving visual "confidence" metrics.

Secondary loss verification executing fundamentally inside the backend testing modules utilized Root Mean Square Error (RMSE) explicitly punishing extreme forecasting outliers heavily enforcing tight local fitting sequences:

$$ \text{RMSE} = \sqrt{\frac{\sum_{t=1}^{n} (\hat{y}_t - y_t)^2}{n}} $$
Equation 5.2: Root Mean Square Error computation strictly mapped for model parameter selection filtering.
